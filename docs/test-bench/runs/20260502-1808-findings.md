## Per-failure detail

### J03/J04 — Deploy lag (Phase G — citizen entry + life-event)

**Failures**: `getByTestId('check-results' | 'program-result-oas' | 'life-event-cta-ei' | 'program-outcome-ei' | 'life-event-ei-result' | 'life-event-no-ei')` not visible.

**Root cause**: Phase G of v3 (citizen entry surface + bounded life-event) shipped in commit `dd5d926` on 2026-04-29. The HF image (`fc19f64` / `b08779c`) predates that. The `data-testid` markers referenced by these tests don't exist in the deployed bundle.

**Action**: leave failing. Will pass automatically when HF rebuilds against v0.5.0.

### J17/J18 — Deploy lag (Phase F — government-leader compare)

**Failures**: `getByTestId('compare-summary' | 'compare-table' | 'compare-exclusion-jp')` not visible on `/compare/ei` and `/compare/oas`.

**Root cause**: Phase F (cross-jurisdiction comparison surface) shipped in commit `924dfc2`, post-image. The `/compare/{programId}` route exists on v2.1 but renders different markup.

**Action**: leave failing. Will pass after HF rebuild.

### J20 — Deploy lag (demo seed contents)

**Failure**: `getByText('demo.draft.ca-oas.age-67-amendment').or(getByText('demo.draft.fr-cnav.indexation-2026'))` not visible.

**Root cause**: The seeded demo drafts on v2.1 carry different keys than v3. The assertion is hardcoded to v3-era seed names.

**Action**: relax the test to assert "queue has at least one draft" rather than specific keys. Promoted to follow-up.

### J24 — Deploy lag + UI behavior on shared deploy

**Failure**: created draft via API, then on `/config/approvals` the page didn't render the new draft within 15s.

**Root cause**: two possibilities:
1. The `/config/approvals` UI on HF v2.1 may filter/paginate differently than v3
2. The SPA route loader cached its draft list at SSR time before our POST landed

**Action**: re-test after HF rebuild. If still failing, refactor to look up the draft via `/api/config/values?status=draft` for the listing assertion (more deterministic than UI).

### J38 — Real backend bug (federation enable, unknown publisher → 500)

**Failure**:
```
POST /api/admin/federation/packs/__test_bench_unknown_publisher__/enable
→ 500 Internal Server Error
```

**Expected**: 404 (publisher not in registry) or 400 (invalid input).

**Root cause**: looking at [`src/govops/api.py:872`](../../src/govops/api.py#L872), the enable endpoint loads the publisher and does not branch on the absent case before calling downstream verification. Stack trace inside HF would confirm the exact line.

**Action**: file as a defect. The failing test stays in the bench as the regression detector.

### J44 — Likely deploy lag (FR cookie at SSR)

**Failure**: `<html>` shows `lang="en"` even when `govops-locale=fr` cookie is set.

**Root cause**: matches the v0.4.0-era ssrLocaleSync bug documented in [`web/e2e/ssr-head.spec.ts`](../../web/e2e/ssr-head.spec.ts) lines 41-50. Fix shipped post-v2.1 image.

**Action**: leave failing; expected to pass after HF rebuild. Same root cause as M02.

### M02 — Likely deploy lag (SSR cookie locale)

**Failure**: `<title>` for `/about` does not contain "propos" when cookie is `govops-locale=fr`.

**Root cause**: same as J44.

**Action**: leave failing; expected to pass after rebuild.

### J03 — Pre-conditions (specific to one assertion in the describe)

The first sub-test (`renders the form with heading + jurisdiction selector`) actually depends on `data-testid="check-form"` which is also v3-era. Same deploy-lag bucket.

## What this baseline run actually tells us

The bench is doing exactly what it's supposed to do:

1. **It runs end-to-end against a live deploy without polluting anything important** — drafts created with `e2e.*` prefixes are clearly synthetic; no production-shape data was touched.

2. **It reports a meaningful version gap** — 7 of 16 failures map directly to "v3 features not yet on HF." That's a useful signal — it means a redeploy of HF to v0.5.0 should close them, and the bench will confirm so.

3. **It found a real defect** — J38 (500 on federation enable for unknown publisher). Worth filing.

4. **It surfaced 6 of my own assumptions that were wrong** — the bench is a forcing function for getting the spec right.

The next run, after HF rebuilds against v0.5.0, will be the second comparable point. The diff between runs is the proof.
