## Two real defects surfaced (not deploy-lag — code defects)

### 1. Dockerfile lost `VITE_API_BASE_URL=""` at build time

**Found in**: bench-4 (after rev2 deploy), root-caused via Playwright trace showing "Could not run the check. Failed to fetch" alert on `/check`.

**Root cause**: commit `35fb187` (v2.1 SPA-build pivot) removed the supervisor script that set `VITE_API_BASE_URL=""` for `vite dev`, and never re-applied it to the build-time `npx vite build` step. The api.ts fallback (`http://127.0.0.1:8000`) was inlined into every visitor's bundle since `35fb187`. Every visitor since then experienced silent fetch failures.

**Fix**: commit `6aa3a47` on main — re-applies `VITE_API_BASE_URL=""` to the `RUN npx vite build` line in stage 1 of the Dockerfile.

**This is the bug the bench actually caught.** Without the bench, this regression would have continued indefinitely — the page rendered, the form rendered, but nothing the visitor did *worked*. The smoke tests passed (page rendered fine); only a journey that exercised an actual API call would catch it.

### 2. Federation enable with unknown publisher returns 500 (J38)

**Found in**: bench-1, persisted across all five runs.

**Root cause**: `POST /api/admin/federation/packs/{unknown}/enable` doesn't gracefully handle the absent-publisher case. Loads the publisher and downstream code throws.

**Action**: file as a defect. The failing test stays as the regression detector.

## What this run proves

1. **The bench works as a deploy-validator.** Five runs in three hours, all stored as comparable artifacts. `diff -u bench-2 bench-5` produces the canonical answer for "what changed."

2. **The bench surfaces real defects, not just framework drift.** The `VITE_API_BASE_URL` regression has been live since `35fb187` (2026-04-29). Without the bench, it would have stayed silent — the page renders, smoke tests pass, only a journey that actually clicks "submit" catches it. That's the point: shallow tests confirm "the page loads"; journey tests confirm "the user's flow completes."

3. **Comparable run records are working as designed.** The journey IDs are stable across runs, the markdown is diff-clean, and the failure classification (deploy-lag vs test-bug vs real-bug) holds up across deploys.
